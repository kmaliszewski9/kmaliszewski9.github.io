I asked myself this question when playing with (de)serialization in Scala-based microservice doing Kafka stream processing. The bill turned out to be quite a rip-off.

We used to use [Circe](https://circe.github.io/circe/) for JSON processing. Arguably, the most popular one, being the standard choice in the modern Scala ecosystem. The library has great functional programming integration and ergonomic API.

Circe is architected around AST (Abstract Syntax Tree). To decode a JSON string, it is parsed to a generic `Json` type and then you build your data structures by traversing it. This gives you flexibility and makes it really convenient to traverse and modify JSON, but of course not for free. Way too often people pay the price without really benefiting from that flexibility.

When you just mostly transform JSON into structures and back you can be much more efficient. How? By moving work to compile time rather than runtime. 

This is the core idea of  https://github.com/plokhotnyuk/jsoniter-scala, one of the fastest libraries on the JVM. It uses Scala macros to generate type-specific reading/writing code at *compile time*, so at runtime there’s no reflective field discovery, no generic “walk the structure” logic, and no codegen warmup.

For example, for `User(name: String, age: Int)` generates something roughly equivalent to:

```scala
new JsonValueCodec[User] {
  private val n0 = "name"
  private val n1 = "age"

  def decodeValue(in: JsonReader, default: User): User = {
    var _name: String = null
    var _age: Int = 0
    if (in.isNextToken('{')) {
      if (!in.isNextToken('}')) {
        in.rollbackToken()
        do {
          val field = in.readKeyAsString()
          if (field == n0)      _name = in.readString(null)
          else if (field == n1) _age = in.readInt()
          else                  in.skip()
        } while (in.isNextToken(','))
        if (!in.isCurrentToken('}')) in.objectEndOrCommaError()
      }
    } else in.readNullOrTokenError(default, '{')
    User(_name, _age)
  }

  def encodeValue(x: User, out: JsonWriter): Unit = {
    out.writeObjectStart()
    out.writeNonEscapedAsciiKey(n0)
    out.writeVal(x.name)
    out.writeNonEscapedAsciiKey(n1)
    out.writeVal(x.age)
    out.writeObjectEnd()
  }
}
```

The key takeaway: there's no reflection at runtime — the macro expands into direct, hand-optimized `JsonReader`/`JsonWriter` calls for each field, which is why it can be so fast.

How fast? [Benchmarks](https://plokhotnyuk.github.io/jsoniter-scala/) show that jsoniter-scala can be anywhere from 2-10x faster depending on the payload and configuration.

I know what you may be thinking - these synthetic benchmarks are detached from reality. I thought the same. Nonetheless, as I found almost 3x speedup on our model, I decided to give it a try. 

The outcome? Less than half the resource consumption, which translated to less than half the required workload size.
